# PhyExpAssistant 项目可行性探索

## 1. 项目目标

PhyExpAssistant 是一个面向物理实验报告自动处理的 Agent 工具，基于开源 Codex 进行二次开发，使用 Git 进行项目管理。

核心目标如下：

- 接收用户输入的实验数据、题目编号及附加选项。
- 调用大语言模型 API，对实验数据进行结构化解析、清洗、计算说明生成和结果组织。
- 按有限题库中的实验题目定义，自动匹配计算逻辑和输出模板。
- 根据用户是否勾选“思考题”“拓展探究”等部分，动态控制输出内容。
- 在需要绘图、拟合、误差分析时，自动与终端或 Python 脚本交互完成计算与作图。
- 最终输出规范化的 `.docx` 实验报告文档。

结论：**项目总体可行，推荐采用“规则计算 + LLM 辅助表达 + 模板生成”的架构，而不是完全依赖 LLM 直接计算。**

---

## 2. 可行性结论

### 2.1 技术上可行

该项目涉及的关键能力均有成熟方案：

- **Agent 框架**：可基于开源 Codex CLI 的交互与工具调用模式进行二开。
- **数据处理**：可用 Python 完成数值计算、误差分析、拟合、插值、绘图等。
- **文档生成**：可使用 `python-docx` 生成 `.docx`，或基于 `.docx` 模板进行替换。
- **模板化输出**：题库和报告结构天然适合“配置驱动”的方案。
- **可选章节输出**：可通过模板开关或渲染上下文控制输出。
- **终端交互**：绘图、拟合、批处理等操作可由 Agent 自动调用本地脚本完成。

### 2.2 主要难点

项目难点不在“能不能做”，而在“如何保证正确率与可维护性”：

- **实验题目多样性**：不同实验的计算逻辑差异较大。
- **数据输入不规范**：用户可能上传文本、表格、截图转录结果或不完整数据。
- **LLM 计算可靠性问题**：直接让大模型做数值计算和误差传播，稳定性不足。
- **模板复杂度**：实验报告可能包含公式、表格、图片、图题、附加部分。
- **题库维护成本**：每个实验题都需要对应的结构定义、计算规则和模板。

### 2.3 推荐原则

为保证质量，建议采用以下原则：

- **LLM 负责理解与组织语言，不负责最终数值真值。**
- **数值计算尽量由 Python 脚本完成。**
- **题目、模板、计算逻辑全部配置化。**
- **生成报告前加入校验层。**
- **先做有限题库 MVP，再逐步扩展题目范围。**

### 2.4 架构审计结论

当前文档的总体方向正确，但架构描述仍偏“功能分层”，对模块边界、数据契约、状态管理和 LLM 权限边界定义不够清晰。后续如果直接按现有描述开发，容易出现以下问题：

- **Agent 层职责过大**：如果让 Agent 自由决定调用脚本、模板和文件路径，系统行为会难以复现。
- **缺少统一数据契约**：输入、计算结果、LLM 上下文、模板字段之间没有单一事实来源，后期容易字段漂移。
- **题库扩展边界不清**：新增实验时可能同时修改核心流程、配置、模板和计算脚本，维护成本会升高。
- **模板与计算耦合**：如果模板字段直接依赖计算脚本内部变量，模板升级会影响计算逻辑。
- **缺少运行产物追踪**：报告、图片、中间 JSON、日志如果没有统一 run id 和 manifest，不利于调试和追责。

因此，建议将架构优化为：**确定性工作流编排 + 领域数据契约 + 实验插件注册 + 外部适配器隔离 + 运行产物可追踪**。

## 3. 推荐总体架构

推荐采用“Clean Architecture / Ports & Adapters”思路，而不是简单把 Codex Agent 放在系统中心。核心原则是：**核心业务不依赖 LLM、不依赖 CLI、不依赖 `.docx` 实现细节**。

推荐拆成以下四个圈层：

1. **领域核心层（Domain Core）**
   - 定义实验、输入、单位、计算结果、模板字段、校验错误等数据模型。
   - 不调用 LLM，不读写文件，不依赖 Codex CLI。
   - 是系统中最稳定、最需要测试覆盖的部分。

2. **应用编排层（Application Workflow）**
   - 实现固定的报告生成流程：解析输入、校验、计算、生成文字、渲染、导出。
   - Agent 只能在预设步骤中调用能力，不应自由拼接任意命令。
   - 每一步输入输出都落到结构化对象，便于调试、复现和回归测试。

3. **实验插件层（Experiment Plugins）**
   - 每个实验以插件形式提供 `spec.yaml`、`compute.py`、模板和测试样例。
   - 核心流程只通过统一接口调用实验插件，不直接关心某个实验的公式细节。
   - 新增实验时优先新增插件，不修改主流程。

4. **基础设施适配层（Adapters）**
   - CLI/Web 输入、LLM API、`.docx` 渲染、文件存储、日志、绘图库都放在适配层。
   - 适配层可以替换，例如从 CLI 换成 Web，从 `python-docx` 换成 `docxtpl`，不影响核心层。

优化后的核心流程如下：

```text
用户输入/文件
  -> Input Adapter
  -> ReportGenerationWorkflow
  -> ExperimentRegistry 加载实验定义
  -> Schema + Unit Normalizer 校验与标准化
  -> Deterministic Compute Plugin 计算与绘图
  -> LLM Adapter 生成说明文字，不改数值
  -> Render Context Validator 校验模板字段
  -> Docx Renderer 生成报告
  -> Artifact Manifest 记录输出、日志和中间结果
```

关键架构约束：

- **LLM 不拥有最终决策权**：LLM 只能生成结构化草稿和说明文字，不能覆盖 Python 计算结果。
- **计算插件是确定性核心**：同一输入、同一配置、同一版本应得到相同计算结果。
- **模板只消费渲染上下文**：模板不得直接读取计算脚本内部变量。
- **所有阶段都可校验**：输入字段、单位、计算结果、模板字段、输出文件都需要校验。
- **每次运行都有 run id**：统一关联输入、中间结果、图片、报告、日志和错误信息。

## 4. 模块设计建议

### 4.1 领域模型与数据契约模块

这是最建议优先补强的模块。它负责定义系统内部的标准对象，避免各模块之间用松散 `dict` 互相传递。

建议至少定义以下对象：

- `ReportRequest`：一次报告生成请求，包含实验编号、学生信息、选项和原始数据引用。
- `ExperimentSpec`：实验定义，来自题库配置，包含输入 schema、模板、计算插件、可选章节。
- `NormalizedInput`：完成字段校验和单位标准化后的实验输入。
- `ComputeResult`：确定性计算结果，包含数值、表格、图片、校验信息。
- `NarrativeResult`：LLM 生成的文字内容，只能引用已锁定的计算结果。
- `RenderContext`：模板唯一可消费的数据结构。
- `ArtifactManifest`：一次运行生成的报告、图片、中间 JSON 和日志索引。

建议在 MVP 阶段使用 `dataclass` 或 `pydantic` 定义这些对象。若希望减少依赖，可以先用 `dataclass + jsonschema`。

### 4.2 输入与标准化模块

负责采集、解析并标准化以下信息：

- 实验题目编号或题目名称。
- 原始实验数据。
- 班级、姓名、学号、实验日期等元信息。
- 是否输出“思考题”“拓展探究”“原始数据附录”等开关。
- 是否需要自动绘图、拟合、误差棒、线性回归等。

推荐输入形式按优先级实现：

1. **JSON**：MVP 首选，最容易校验和复现。
2. **CSV/Excel**：适合表格型实验数据。
3. **命令行问答输入**：适合补充缺失字段。
4. **自然语言输入**：由 LLM 转为结构化草稿，但必须再经过 schema 校验。
5. **Web 表单**：后续扩展，不应影响核心流程。

该模块需要内置单位标准化能力，例如 `cm -> m`、`ms -> s`，并在校验结果中明确记录自动转换。

### 4.3 实验注册与题库模块

题库建议做成**配置文件 + 计算插件 + 文档模板 + 样例测试**四件套：

- `spec.yaml` 描述实验字段、单位、模板、输出结构、可选章节和 LLM 任务。
- `compute.py` 负责该实验的确定性计算。
- `.docx` 模板负责最终版式。
- `fixtures/` 保存样例输入和期望输出，用于回归测试。

核心系统通过 `ExperimentRegistry` 加载实验，不直接 import 所有实验模块。新增实验的理想路径是：新增一个实验目录并注册配置，而不是改动主流程代码。

### 4.4 应用编排与 Agent 模块

该模块是“固定状态机”，不是开放式自动代理。建议将流程收敛为明确阶段：

- `load_request`
- `resolve_experiment`
- `normalize_input`
- `validate_input`
- `compute_result`
- `generate_narrative`
- `build_render_context`
- `render_docx`
- `write_manifest`

Codex/LLM 可以辅助执行其中部分阶段，但不能绕过阶段校验。这样做能降低 Agent 不可控行为，也方便未来切换到 Web 后端或批处理服务。

### 4.5 计算与绘图模块

建议由 Python 实现，职责包括：

- 基础统计量计算。
- 平均值、标准差、不确定度计算。
- 误差传播。
- 线性/非线性拟合。
- 数据平滑、插值。
- 图像绘制与导出。
- 生成中间结构化结果供模板渲染。

建议把公共能力放入 `compute/`，把实验特有公式放入对应实验插件。常用库建议：

- `numpy`
- `scipy`
- `pandas`
- `matplotlib`
- `sympy`
- 可选：`openpyxl`、`seaborn`、`statsmodels`

### 4.6 LLM 适配模块

LLM 适合承担以下职责：

- 理解用户自然语言输入。
- 将非结构化实验数据转为结构化数据草稿。
- 对结果生成“实验结论”“误差分析”“思考题回答”的文字初稿。
- 对缺失信息生成追问。

不建议让 LLM 直接承担：

- 最终数值计算。
- 拟合参数真值输出。
- 图像数值读取的最终定量结果。
- 文件路径、模板路径、计算模块路径等执行级决策。

LLM 输出建议统一要求为结构化 JSON，并在进入下一阶段前做校验。对于包含数字的文字内容，应增加“数值一致性检查”，确保文字中的结果与 `ComputeResult` 一致。

### 4.7 文档生成模块

推荐两种思路：

#### 方案 A：直接使用 `python-docx` 生成

优点：

- 可完全程序化控制。
- 便于插入表格、图片、段落。

缺点：

- 模板样式维护相对麻烦。

#### 方案 B：使用 `.docx` 模板 + 占位符渲染

优点：

- 更适合固定格式实验报告。
- 非程序人员也可维护模板版式。

缺点：

- 复杂表格和条件块替换实现稍复杂。

**推荐优先采用方案 B，但需要增加模板字段契约校验。**

渲染模块只接收 `RenderContext`，不直接调用计算模块或 LLM。渲染前应检查：

- 模板中所有占位符都能在 `RenderContext` 中找到。
- `RenderContext` 没有未被模板消费的关键字段。
- 图片路径存在且格式符合要求。
- 条件块使用的开关字段存在且类型正确。

### 4.8 存储、日志与产物管理模块

建议为每次运行创建独立目录，例如：

```text
data/outputs/2025-01-01_exp_001_8f3a2c/
├─ input.normalized.json
├─ compute.result.json
├─ narrative.result.json
├─ render.context.json
├─ figures/
├─ report.docx
└─ manifest.json
```

`manifest.json` 应记录输入文件、实验版本、模板版本、模型配置摘要、生成时间、输出文件和错误信息。这样可以支撑问题复现、结果追踪和后续批处理。

## 6. 需要准备的核心文件

要让系统真正可运行，至少需要准备以下几类文件。

### 6.1 全局配置文件

建议文件：`configs/app.yaml`

用途：

- 定义输出目录。
- 定义默认渲染器。
- 定义是否启用绘图。
- 定义日志等级。
- 定义文档导出策略。

建议字段：

```yaml
app_name: PhyExpAssistant
output_dir: data/outputs
default_renderer: docx_template
enable_plot: true
language: zh-CN
log_level: INFO
artifact_policy:
  save_intermediate_json: true
  save_manifest: true
  run_id_format: "{date}_{experiment_id}_{hash}"
```

### 6.2 模型配置文件

建议文件：`configs/model.yaml`

用途：

- 配置 API 地址、模型名、密钥读取方式、超参数。

建议字段：

```yaml
provider: openai_compatible
base_url: ${MODEL_BASE_URL}
api_key: ${MODEL_API_KEY}
model: gpt-4.1
temperature: 0.2
max_tokens: 4000
timeout_seconds: 120
retry:
  max_attempts: 2
  backoff_seconds: 3
guardrails:
  allow_numeric_override: false
  require_json_output: true
```

### 6.3 题目配置文件

建议每道实验题对应一个 `spec.yaml` 文件，如：`src/phyexpassistant/experiments/exp_001/spec.yaml`

用途：

- 描述该实验需要哪些输入。
- 指定使用哪个计算入口和哪个模板。
- 指定可选章节。
- 指定输出字段。

建议格式：

```yaml
id: exp_001
version: 1
name: 单摆测重力加速度
category: mechanics
compute:
  module: phyexpassistant.experiments.exp_001.compute
  entrypoint: run
renderer:
  engine: docx_template
  template: src/phyexpassistant/experiments/exp_001/templates/report.docx
unit_policy:
  system: SI
  accepted_units:
    length: [m, cm, mm]
    period: [s, ms]
plot:
  enabled: true
  outputs:
    - key: plot_fit
      filename: pendulum_fit.png
input_schema:
  fields:
    length:
      type: number_array
      unit: m
      required: true
    period:
      type: number_array
      unit: s
      required: true
    room_temperature:
      type: number
      unit: celsius
      required: false
report_sections:
  basic:
    - experiment_purpose
    - experiment_principle
    - raw_data
    - calculation
    - result
  optional:
    thinking:
      enabled_by: include_thinking
    extension:
      enabled_by: include_extension
outputs:
  metrics:
    - key: g_mean
      unit: m/s^2
      precision: 3
    - key: g_uncertainty
      unit: m/s^2
      precision: 3
render_contract:
  required_fields:
    - student_name
    - experiment_name
    - raw_data_table
    - calc_result_table
    - result_summary
    - error_analysis
    - g_mean
    - g_uncertainty
    - plot_fit
llm_tasks:
  - key: result_summary
    prompt: result_summary.txt
    source: compute_result
  - key: error_analysis
    prompt: error_analysis.txt
    source: compute_result
fixtures:
  input: fixtures/input.json
  expected: fixtures/expected.json
```

### 6.4 输入数据格式文件

建议至少支持 JSON 和 CSV。

#### JSON 示例

```json
{
  "experiment_id": "exp_001",
  "student": {
    "name": "张三",
    "student_id": "20250001",
    "class_name": "物理实验一班"
  },
  "options": {
    "include_thinking": true,
    "include_extension": false,
    "include_raw_appendix": true
  },
  "data": {
    "length": {
      "unit": "m",
      "values": [0.5, 0.6, 0.7]
    },
    "period": {
      "unit": "s",
      "values": [1.42, 1.55, 1.67]
    }
  }
}
```

#### CSV 示例

适用于表格型实验数据，推荐首行是字段名。字段名可以使用带单位后缀的别名，由标准化模块映射回 `spec.yaml` 中的字段，例如：

```csv
length_m,period_s
0.5,1.42
0.6,1.55
0.7,1.67
```

### 6.5 Prompt 模板文件

建议将 Prompt 单独存放，避免写死在代码里。

例如：`src/phyexpassistant/prompts/error_analysis.txt`

用途：

- 根据计算结果自动生成误差分析。
- 限制语气和篇幅。
- 限制模型不得编造数值。

建议 Prompt 特征：

- 明确输入字段。
- 明确输出格式。
- 明确“禁止修改计算结果”。
- 明确语言风格为实验报告体。

### 6.6 `.docx` 报告模板文件

这是项目的关键资产，建议分为：

- **通用模板**：适合大部分实验。
- **题目专用模板**：针对特殊表格、特殊图像布局。

---

## 7. 模板格式建议

### 7.1 文档模板推荐形式

建议使用 `.docx` 文件作为母版，在其中预留占位符。

例如：

- `{{student_name}}`
- `{{student_id}}`
- `{{class_name}}`
- `{{experiment_name}}`
- `{{experiment_date}}`
- `{{result_summary}}`
- `{{error_analysis}}`

对于图片占位建议使用语义字段：

- `{{plot_main}}`
- `{{plot_fit}}`

对于表格数据建议在模板规范中约定：

- 某个表格整体由程序重建。
- 或在模板中仅保留表题，内容由程序插入。

### 7.2 可选章节控制

“思考题”“拓展探究”等建议不要写死在主流程中，而是使用**条件渲染**机制。

有两种实现方向：

#### 方向 A：使用多个模板

- 基础模板
- 含思考题模板
- 含拓展模板

优点：实现简单。

缺点：模板数量会快速膨胀。

#### 方向 B：单模板 + 条件块标记

例如约定以下逻辑标记：

```text
[[IF include_thinking]]
思考题：{{thinking_answer}}
[[END_IF]]
```

```text
[[IF include_extension]]
拓展探究：{{extension_content}}
[[END_IF]]
```

**推荐方向 B。**

注意：`.docx` 本身不是纯文本，直接在 Word 中做复杂条件块替换会有实现成本。实际工程中可选：

- 先用中间模板语言（如 Markdown/Jinja）渲染，再转 `.docx`。
- 或自己实现简化版占位符和条件块解析器。

### 7.3 模板字段命名规范

建议统一采用蛇形命名：

- `student_name`
- `experiment_id`
- `experiment_name`
- `raw_data_table`
- `calc_result_table`
- `result_summary`
- `thinking_answer`

统一规范的好处：

- 减少模板与代码耦合。
- 方便做字段校验。
- 便于后续增加题目。

---

## 8. 计算与绘图的推荐实现方式

对于实验类项目，推荐采用“公共计算库 + 每题一个计算插件”的策略。

例如：

- `src/phyexpassistant/compute/uncertainty.py`
- `src/phyexpassistant/compute/fitting.py`
- `src/phyexpassistant/experiments/exp_001/compute.py`
- `src/phyexpassistant/experiments/exp_002/compute.py`

每个实验插件对外提供统一接口，例如：

```python
from phyexpassistant.domain.models import ComputeResult, NormalizedInput, RunContext


def run(input_data: NormalizedInput, context: RunContext) -> ComputeResult:
    ...
```

`run()` 的约束建议如下：

- 只接收已经通过 schema 和单位标准化的数据。
- 不直接读取用户原始文件。
- 不直接调用 LLM。
- 不直接渲染 `.docx`。
- 所有图像写入 `RunContext` 指定的产物目录。
- 返回结构化 `ComputeResult`，不要返回松散 `dict`。

返回结构建议包含：

- `metrics`：带单位、有效数字和来源说明的计算指标。
- `tables`：标准化表格数据，供模板渲染器重建表格。
- `figures`：图像路径、图题、引用 key。
- `checks`：物理合理性检查、异常值检查、单位转换记录。
- `llm_context`：给 LLM 的只读摘要，不包含可修改的最终数值。
- `render_fields`：可直接进入 `RenderContext` 的字段。

例如：

```json
{
  "metrics": {
    "g_mean": {
      "value": 9.79,
      "unit": "m/s^2",
      "precision": 3,
      "source": "least_squares_fit"
    },
    "g_uncertainty": {
      "value": 0.08,
      "unit": "m/s^2",
      "precision": 2,
      "source": "uncertainty_propagation"
    }
  },
  "figures": {
    "plot_fit": {
      "path": "data/outputs/2025-01-01_exp_001_8f3a2c/figures/pendulum_fit.png",
      "caption": "周期平方与摆长的线性拟合图"
    }
  },
  "checks": [
    {
      "level": "warning",
      "code": "SMALL_SAMPLE_SIZE",
      "message": "周期测量组数少于 5，建议增加重复测量。"
    }
  ],
  "render_fields": {
    "g_mean": "9.790 m/s²",
    "g_uncertainty": "0.08 m/s²",
    "plot_fit": "data/outputs/2025-01-01_exp_001_8f3a2c/figures/pendulum_fit.png"
  }
}
```

计算模块的测试重点应覆盖：公式正确性、单位转换、异常输入、边界值、图像文件生成和渲染字段完整性。

## 9. Agent 工作流建议

建议 Agent 工作流固化为可复现的状态机：

1. 创建 `run_id` 和产物目录。
2. 读取用户输入或上传文件，生成 `ReportRequest`。
3. 解析实验编号；无法确定时只允许追问用户确认。
4. 通过 `ExperimentRegistry` 加载 `ExperimentSpec`。
5. 如输入为自然语言，调用 LLM 生成结构化草稿。
6. 执行 schema 校验、单位标准化和缺失字段检查。
7. 若校验失败，输出明确问题并追问，不进入计算。
8. 调用实验 `compute.run()` 生成 `ComputeResult`。
9. 执行计算结果校验，包括数值范围、单位、图像文件、必需指标。
10. 将锁定后的 `ComputeResult` 交给 LLM 生成 `NarrativeResult`。
11. 校验 LLM 文本中的数值是否与 `ComputeResult` 一致。
12. 合并 `ComputeResult` 与 `NarrativeResult`，生成 `RenderContext`。
13. 校验模板占位符和 `RenderContext` 的字段契约。
14. 调用 `.docx` 渲染器生成报告。
15. 写入 `manifest.json`，输出报告路径和日志摘要。

该流程中，**Workflow 是确定性主线，计算模块是数值真值来源，LLM 是受控辅助节点**。

建议把每个阶段都设计为可单独测试的函数。这样 CLI、批处理和 Web 后端都可以复用同一条应用工作流。

## 10. 风险与规避建议

### 10.1 Agent 权限失控风险

风险：Agent 自由选择命令、路径、模板和计算脚本，导致结果不可复现或误调用非目标文件。

规避：

- 将 Agent 收敛为固定 Workflow。
- 文件路径只能来自配置和 `RunContext`。
- LLM 不允许决定计算模块路径、模板路径和输出路径。
- 工具调用前后都记录到 `manifest.json`。

### 10.2 数据契约漂移风险

风险：输入字段、计算结果字段、Prompt 字段和模板字段分别演化，导致后期频繁生成失败。

规避：

- 以 `ExperimentSpec` 和 `RenderContext` 作为单一事实来源。
- 启动时校验实验配置、计算插件和模板字段。
- 每个实验维护样例输入和期望输出。
- 模板字段变更必须同步更新 `render_contract`。

### 10.3 数值错误风险

风险：LLM 输出与真实计算不一致。

规避：

- 所有最终数值来自 Python。
- LLM 提示词中明确禁止修改计算结果。
- 对 LLM 生成文本做数值一致性检查。
- 关键实验增加单元测试。
- 对计算结果增加物理范围和单位检查。

### 10.4 模板维护风险

风险：模板字段名变动导致生成失败。

规避：

- 建立模板字段清单。
- 启动时校验模板所需字段。
- 给每个模板加版本号。
- 将复杂表格交给渲染器重建，不在 Word 中手工维护大量占位符。

### 10.5 输入脏数据风险

风险：用户输入单位混乱、漏填、格式不统一。

规避：

- 引入输入 schema 校验。
- 对单位做标准化转换。
- 对缺失值进行追问，不自动猜测。
- 在中间 JSON 中记录原始值、标准化值和转换规则。

### 10.6 题库扩展成本

风险：每新增一个实验都要改代码。

规避：

- 把公共计算逻辑抽到通用模块。
- 将差异部分放在 `spec.yaml`、实验 `compute.py` 与专用模板中。
- 通过 `ExperimentRegistry` 管理实验插件。

### 10.7 隐私与日志风险

风险：学生姓名、学号、实验数据或 API Key 被写入日志、Prompt 或产物包。

规避：

- API Key 只能从环境变量读取。
- 日志中默认脱敏姓名、学号和密钥。
- `manifest.json` 只记录模型配置摘要，不记录密钥。
- 如果接入云端 LLM，应明确哪些字段会被发送给模型。

### 10.8 可复现性风险

风险：同一输入在不同依赖版本、模型版本或模板版本下输出不一致。

规避：

- 固定 Python 依赖版本。
- 在 `manifest.json` 记录实验版本、模板版本、代码版本和模型名称。
- 对计算结果使用确定性随机种子。
- 将中间 JSON 作为调试和回归测试依据。

## 11. MVP 建议范围

建议第一阶段只做一个最小可用版本：

- 仅支持 **1~2 个典型实验题目**，先打通插件规范和端到端链路。
- 仅支持 **JSON/CSV 输入**，自然语言输入只作为结构化草稿生成能力。
- 仅支持 **本地命令行调用**。
- 仅支持 **固定 `.docx` 模板渲染**。
- 仅实现 **思考题开关**，暂不做太复杂的条件块。
- 仅接入 **一个 LLM API 提供方**。
- 必须实现 **运行产物目录和 `manifest.json`**。
- 必须实现 **输入 schema 校验、单位标准化、模板字段校验**。

MVP 阶段建议明确不做：

- 不做 OCR 截图识别。
- 不做 Web 平台。
- 不做多模型路由。
- 不做任意实验自动推导公式。
- 不让 LLM 直接生成最终数值结果。

推荐优先选择的实验类型：

- 线性关系明显、便于拟合的实验。
- 数据表结构固定的实验。
- 报告模板相对统一的实验。

建议从一开始就按 Git 工程化方式组织：

- `main`：稳定可用版本。
- `dev`：日常开发分支。
- `feature/*`：功能分支，如 `feature/template-render`。

## 14. 总结

PhyExpAssistant 非常适合作为一个“有限题库 + 模板化报告 + 计算脚本 + LLM 辅助表达”的 Agent 项目。

最关键的工程决策是：

- **用固定 Workflow 约束 Agent 行为。**
- **用领域数据契约打通输入、计算、LLM 和模板。**
- **用实验插件管理题库差异。**
- **用 Python 保证计算正确性。**
- **用 LLM 提升输入理解和报告文字生成能力，但不让它决定最终数值。**
- **用 `.docx` 模板和字段契约保证最终报告格式稳定。**
- **用 `manifest.json` 保证每次运行可追踪、可调试、可复现。**

因此，本项目建议按 **MVP -> 小规模题库扩展 -> 通用化平台** 的节奏推进，而不是一开始就追求全题型覆盖。
